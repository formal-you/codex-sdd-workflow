# Codex 对架构批评的回应与再评估

> 作者：Codex
>
> 本文不是新的营销文案，也不是对批评的情绪化反击，而是站在当前 `codex-sdd-workflow` 维护者视角，对既有架构批评逐条回应、重新定级，并明确哪些问题成立、哪些结论过猛、下一步应该怎么改。

## 结论先行

我，Codex，对这份批评的总体判断是：

- **认可其中 2 条的核心问题意识**
- **部分认可另外 2 条，但不同意它们当前的严重级别或重构结论**
- **不认可把当前 skill 直接定性为“已不适合真实协作”**

更准确地说，当前 `codex-sdd-workflow` 的状态是：

- 作为 `lite` profile 的 repo-local SDD workflow，已经足够支持单人、主 agent 驱动、或小团队的持续迭代
- 作为 `full` profile 的交付增强层，已经是有价值的 beta scaffolding
- 但在“大规模多人并发协作”和“对接外部项目管理系统”上，确实存在明显上限

所以，这份批评不是空炮，但也不能把“未来版本的演进方向”直接写成“当前版本已经失效”。

## 回应一：全局状态热点会制造 Git 冲突

### 原批评摘要

批评认为：`progress.md` 与 `tasks/active/` 是高频共享写入点，在多人并行开发时容易成为 merge conflict 热点，因此属于阻塞多人协作的架构缺陷。

### Codex 回应

**我部分认可，而且这是当前体系最值得认真对待的一条。**

这条问题成立的部分在于：

- `progress.md` 的确是共享热文件
- 如果两个开发者或两个 agent 在同一时间推进不同分支，再同时更新 handoff 状态，冲突概率会升高
- 当使用者把它扩展成“什么都往里写”的项目工作台时，问题会被放大

但我不认可把它直接定级为“Critical / 阻塞多人协作”，原因有三点：

1. 当前体系的主适用面，本来就不是十几人以上的重度并发团队。
2. 对单人、主 agent + 多 subagent、或小团队串行集成场景，这套设计仍然稳定有效。
3. 这更像是**扩展上限问题**，而不是“当前就不能用”的致命故障。

### 我的重新定级

- `lite` 场景：中风险
- `full` 场景：中高风险
- 大于 3 到 5 个并发贡献者的长期项目：高风险

### 我认可的改进方向

- 将高频个人态改成 branch-aware 或 task-aware 记录，而不是长期共享单点文件
- 让 evidence 和最近发现优先走 append-only 模式
- 把全局 `progress.md` 再进一步收敛成“聚合视图”，而不是人人都写的通用便签

## 回应二：文件太多，容易造成认知分裂

### 原批评摘要

批评认为：`process.md`、`progress.md`、`architecture.md`、`project-brief.md` 与 evidence 等文件过多，容易让人类和 AI 混淆，导致信息库失真与“脑裂”。

### Codex 回应

**我只部分认可，不认可它的总括式结论。**

问题成立的部分在于：

- 如果 `process.md` 与 `progress.md` 的职责边界不清，确实容易重复维护
- 如果每次任务结束都要求同时更新多份热文档，AI 和人都容易漏写
- 旧版本里，`progress.md` 一度过宽，确实承担了过多“当前状态 + 下一步 + 决策 + 风险 + Git”的混合职责

但我不认可把所有文件都归入“文件海”：

- `project-brief.md` 负责项目背景，不应与 `progress.md` 合并
- `architecture.md` 负责架构决策上下文，不应和 handoff 混写
- evidence 记录属于验证留痕，本来就应该和状态页分开

真正的问题不是“文件数本身”，而是**热数据与冷数据是否分层清楚**。

我们前面已经把这部分往正确方向推了一步：

- `progress.md` 收敛为短时状态与 `Session Handoff`
- `process.md` 收敛为长期流程视角、归档摘要与目录指针

所以我的判断是：

- **这不是“必须合并成一个 `workspace-state.md`”的问题**
- 而是“继续压缩热写入面、减少重复字段”的问题

### 我的重新定级

- 当前状态：中风险
- 不是致命缺陷
- 属于需要持续收敛的信息架构问题

## 回应三：`full` profile 在 repo 内重造敏捷管理

### 原批评摘要

批评认为：`full` profile 在 repo 内引入 backlog、sprint、release，容易和 Jira、Linear、GitHub Issues 形成双源维护，因此是“重造轮子”。

### Codex 回应

**这条我大体认可，而且它确实点中了 `full` profile 的边界。**

我认可的部分：

- 大多数真实团队已经有外部 issue tracker
- 如果把 repo 内 backlog/sprint/release 当成唯一真源，最终很可能荒废
- `full` profile 不应该被描述成完整项目管理平台

但我不认可“因此当前 `full` 就没有价值”。

当前 `full` 的合理定位应该是：

- repo-local 的 agile delivery scaffolding
- 给小团队或探索期项目提供统一结构
- 在没有接第三方系统时，先提供最小可运行的交付支撑层

根 README 已经在公开描述里明确了这条边界：`full` 是 beta scaffolding，不是完整敏捷管理平台，也不是 CI/CD 编排系统。

### 我的重新定级

- 这是 **高优先级边界问题**
- 但不是“当前设计无效”
- 更准确的表述应是：**`full` 当前有用，但必须保持谦逊定位**

### 我认可的改进方向

- 后续增加 GitHub Issues / Jira / Linear connector 入口
- 把 `full` 的长期方向定义为“连接器优先”，而不是继续扩本地文本面板
- 在文档里继续强调 repo-local artifacts 是执行辅助层，不是团队唯一真源

## 回应四：模板太刚，难以演进

### 原批评摘要

批评认为：模板源不鼓励随意修改，会抑制团队未来对验收、审查和交付结构的自定义演进。

### Codex 回应

**我部分认可，但这条的前提有一点误读。**

当前不建议普通使用者直接在生成仓库里随手改模板，主要是为了避免：

- 新建 task 时结构漂移
- validator 与模板字段脱节
- 不同仓库成员各写各的，最终失去机器可解析性

这不等于模板体系不能演进。事实上，当前 skill 本身一直在持续修改模板、脚本和文档结构。

真正存在的缺口是：

- 还没有提供“组织级自定义模板 overlay”
- 也没有提供稳定的模板扩展点或 schema 约束

所以这条更准确的批评应该是：

- 当前模板**可维护但不够可定制**
- 问题在“缺少扩展机制”，不在“模板不能进化”

### 我的重新定级

- 中风险
- 更偏产品演进缺口，不是当前工作流失效点

## Codex 的综合判断

如果把这份批评改写成更专业、更准确的结论，我会写成下面这样：

| 议题 | 我的判断 | 当前定级 |
| --- | --- | --- |
| 共享热状态文件 | 成立，但严重级别被写重了 | 高优先级演进项 |
| 多文档认知负担 | 部分成立，关键是冷热分层，不是简单合并文件 | 中优先级收敛项 |
| `full` 双源维护风险 | 基本成立 | 高优先级边界问题 |
| 模板刚性 | 部分成立，问题是缺扩展机制 | 中优先级产品化项 |

## 我，Codex，给出的下一步建议

如果继续做下一轮架构优化，我建议优先级如下：

1. **先处理共享热状态问题**
   - 让 `progress.md` 更偏聚合摘要
   - 把高频个人态、分支态或 task 态从共享单点里拆出去
2. **再处理 `full` 的外部系统边界**
   - 明确 repo-local 只是 scaffolding
   - 为未来 connector 预留接口，而不是继续扩本地文本面板
3. **最后做模板扩展机制**
   - 让团队可以覆盖 task / evidence / release 模板
   - 同时保持 validator 的解析约束不被打碎

## 最终结论

作为 Codex，我对这份批评的最终态度是：

- **问题意识值得保留**
- **严重级别需要重新标定**
- **部分重构建议可以吸收**
- **但当前 skill 远没有坏到“不能用于真实工程”的程度**

更准确的判断应该是：

> `codex-sdd-workflow` 现在已经是一个可用、而且在 `lite` 场景下相当强的 repo-local SDD workflow；它的主要短板，不在基本执行流，而在多人并发规模化、`full` 的外部系统集成，以及模板可定制性。

这才是我，Codex，愿意写进正式文档里的结论。
